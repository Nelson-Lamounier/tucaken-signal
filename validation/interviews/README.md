# Engineer interviews — AI-transparency pillar validation

The single piece of v1.0 validation that cannot be automated: 5 real
engineers giving real reactions to the AI-transparency framing before
the pillar ships at default-on weight.

This directory contains everything you need to run the interviews
without re-inventing the protocol mid-conversation.

## Files

- `ai-transparency-question-set.md` — the actual questions (already
  written). Don't change unless you want to invalidate prior runs.
- `outreach-email.md` — copy-paste-ready outreach to send to candidates.
- `scheduling.md` — calendar slot template + Calendly link placeholder.
- `mock-ai-usage-doc.md` — the artifact shown to interviewees in Q4
  ("react to this draft AI_USAGE.md"). Generated by tucaken-signal
  against ai-applications.
- `responses/` — one markdown file per interview, using
  `responses/_template.md` as the structure.
- `analysis.md` — after all 5 interviews, fill this with the synthesis
  + the decision gate result.

## Protocol

1. **Recruit 5 engineers.** Target mix:
   - 2 juniors (0–2 yrs)
   - 2 mid (2–5 yrs)
   - 1 senior (5+)
   - Mix of AI-heavy and AI-light coders. Avoid asking only your own
     team — bias risk.
2. **Send outreach** (`outreach-email.md`) with the question set
   attached and a 30-min slot booking link.
3. **Run each interview live** (Zoom / Meet / in-person). 30 min, follow
   the question set, capture answers in `responses/<initials>.md`. Do
   not paraphrase live — read questions verbatim, transcribe near-verbatim.
4. **After all 5**, fill `analysis.md` with patterns, the decision gate
   outcome, and the resulting product change (if any).

## Decision gate (from `ai-transparency-question-set.md`)

- ≥3 of 5 say it feels like compliance overhead → re-frame the pillar
  as opt-in suggestion, not part of default scoring penalty.
- Mixed (2–2–1) → keep as recommendation, no score deduction.
- ≥3 see it as credibility win → keep current framing AND promote to
  default-visible suggestion when no doc exists.

## Why this can't be automated

The question is sociological, not technical. The framing of
"transparent AI disclosure as credibility signal" assumes a specific
2026 hiring-norm hypothesis. If engineers reading the suggestion
react with "this feels like the new corporate compliance form,"
the pillar's whole framing is wrong and shipping it default-on will
hurt the product's credibility, not help it.

The static signals (`HumanJudgmentDetector`, `AiUsageAnalyzer`) are
already shipped and tested. What's untested is whether the *output*
of those signals — the recommendation text presented to the engineer
— lands the way the spec intends. Only humans answering honestly
about their own reactions can validate that.

## Time budget

- ~30 min per interview × 5 = 2.5 hours of interview time
- ~30 min outreach + scheduling
- ~1 hour synthesis + analysis

**Total: ~4 hours over 1–2 weeks calendar time.** This is the only
remaining v1.0 blocker that requires you (and only you) to do.
