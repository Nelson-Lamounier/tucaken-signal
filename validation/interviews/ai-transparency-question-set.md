# AI-transparency pillar — engineer interview question set

Goal: validate whether engineers will actually adopt `AI_USAGE.md`
documentation, or whether it feels like compliance overhead.

## Recruit

5 engineers, mix of:
- 2 juniors (0–2 yrs)
- 2 mid (2–5 yrs)
- 1 senior (5+)

Mix of AI-heavy and AI-light coders.

## Questions

1. When you look at someone else's GitHub project in 2026, do you
   try to figure out how much AI they used? Why or why not?
2. If a portfolio repo included an `AI_USAGE.md` documenting which
   parts used AI, would that *increase* or *decrease* your trust in
   the engineer? Walk through your reasoning.
3. Have you written anything like this for your own projects?
   Why / why not?
4. Show this mock `AI_USAGE.md` (provide the sample). Reaction —
   is it useful, performative, or risky?
5. If a tool generated a draft `AI_USAGE.md` for you, would you
   keep it, edit it, delete it, or ignore the suggestion?
6. What would *increase* the chance you actually use this?
   (e.g. brevity, tone, format, framing)

## Decision gate

- If ≥3 say it feels like overhead or compliance noise:
  re-frame as opt-in suggestion (not part of default scoring penalty).
- If reaction is mixed (2–2–1): keep current framing; surface as
  recommendation only, not a score deduction.
- If ≥3 see it as a credibility win: keep current framing AND
  promote it to a default-visible suggestion when no doc exists.
